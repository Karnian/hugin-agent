## 1. What Hugin Agent is

An **outbound-only local bridge daemon** (`hugind`). It installs on a user's
machine behind NAT/firewall, **dials out** over WSS to a cloud orchestrator,
receives commands, runs the local **Claude Code / Codex CLI headlessly**, and
streams normalized results back. No inbound port is ever opened. The cloud relay
is a separate effort; **this repo owns the local side + the shared wire protocol.**

The protocol is a **shared contract** — both `hugind` and the cloud relay import
the same zod SSOT. Changing wire-visible fields requires both sides to agree.

## 4. Locked decisions (do NOT relitigate)

- **EventKind = Option A: core kinds are FROZEN for v1.** Engine-specific events
  use the `vendor.<engine>.*` namespace only. A new *core* kind requires a major
  bump (v2). This makes "unknown core kind → NACK" (F6) consistent: within v1,
  cores never grow, so an unknown core means a bug/wrong version.
- **`agent_id` = per-device**, with a **user 1:N agent** layer above it:
  `tenant_id > user_id > agent_id (one device/daemon) > engines[]`.
  - "One user, many CLIs" is covered two ways, both by per-device: many machines
    → many `agent_id` under one `user_id`; one machine running claude+codex →
    one `agent_id` with multiple `capabilities.engines`.
  - `connection_epoch` and lease ownership are scoped **per `agent_id`**, so one
    device reconnecting never fences another device of the same user.
  - **Wire does NOT change for this.** `agent_id` stays per-device; `user_id`
    lives one level up in the pairing record (it groups `agent_id`s). The signed
    transcript keeps `tenant_id` only (cloud decision) — `agent_id` represents
    the (user, device) pair. Document `user_id` in the auth-spec identity model
    + audit, not on the wire.

### 5b. EventKind A + F6 rule (`messages.ts` + `protocol/README.md`)

- `EventKind` already rejects unknown core kinds at parse time (closed enum) —
  that IS the F6 enforcement. **Do not add a `non-critical` field** (Option A
  means cores are frozen, so the "ignore non-critical unknown core" branch
  doesn't exist in v1). Just **document** the rule in README:
  - Unknown **core** kind → receiver NACKs (`invalid_message`); it must NOT be
    reinterpreted as vendor.
  - Unknown **vendor** kind → may be ignored/logged/passed through opaquely.
  - **Sender obligation:** a sender only emits core kinds defined at/below the
    negotiated version. (Within frozen-v1 this is automatic.)
- Note: `validateInbound()` can't do F6 — `parseMessage()` throws first. F6 lives
  in the parse→nack translation layer the daemon/relay implement; document it.

### 5c. Spec edits — auth/pairing (`docs/auth-pairing-spec.md`)

- **F2 server_origin normalization — finish the edge cases** (it's signed, so it
  must be byte-identical on both sides). Add, normatively:
  - production = DNS hostname only; **reject bare IPv4/IPv6** for tenant origins.
  - local-dev exception explicitly limited to `localhost` / `127.0.0.0/8` /
    `[::1]`, and only there may `ws://` be allowed.
  - **reject port 0**; omit default ports (`:443`/`:80`); IPv6 bracket form
    `[::1]`; reject zone-IDs and percent-encoded authority; force lowercase even
    after IDNA punycode output.
- **F3** — `tenant_id` grammar `1*128(ALPHA/DIGIT/"-"/"_"/".")` + key-lifetime
  stability + no silent cross-tenant rebind (re-pairing required). Cross-check
  the `AuthId` type from 5a covers `challenge_id`/`agent_id`/`key_id`.
- **F5 linearizability — make concrete.** The POP MUST linearize: (1) nonce
  consume (atomic check-unspent→mark-spent), (2) `connection_epoch` issuance
  (strictly monotonic **per `agent_id`**), (3) lease ownership (≤1 owner outside
  the rotation overlap, CAS/fencing). OK: strong SQL / etcd / Consul / ZK /
  FoundationDB / Dynamo conditional writes. NOT OK: eventually-consistent cache,
  per-region Redis, pod-local memory. Note re-pairing (new `agent_id`) may reset
  the epoch namespace. Cross-POP stays out of MVP scope.
- **Identity model** — add the `tenant_id > user_id > agent_id(device) > engines`
  hierarchy and the per-device decision (§4). Resolve the open "agent_id
  per-device vs per-host" item to **per-device**.
- **Clarifications:** all base64url is unpadded (reject `=`); `server_origin` /
  `tenant_id` are NOT in `hello` (verifier reconstructs them from the pairing
  record + trusted connection metadata; reject auth if not reconstructable);
  transcript `protocol_version` is the exact value in `hello` (don't verify under
  a different negotiated version — reject); `challenge_ttl_ms` is advisory.

### 5d. NEW — F4 cross-language test vectors (`protocol/v1/test-vectors.json` + generator)

Goal: guarantee a TS daemon and a Go/Rust relay compute **byte-identical**
transcripts + signatures. Write a small generator (`tsx`, Node `crypto` Ed25519)
that emits golden vectors and a checker the selftest can run.

Each positive vector includes:
`{ challenge_id, nonce_base64url, nonce_raw_hex, agent_id, key_id,
protocol_version, tenant_id, input_server_origin, canonical_server_origin,
domain_tag_hex (LP("hugin-agent/auth/v1")), transcript_hex, ed25519_seed_hex,
ed25519_public_hex, expected_signature_base64url }`.

Transcript construction is auth-spec §5: `LP(x)=uint32_be(len)||x`, with
`nonce_raw` inserted as the **raw 32 bytes (no LP)**. Include negative/edge
vectors that MUST fail: 42- and 44-char nonce, padded nonce (`…==`), uppercase
host, `:443` present, trailing dot, an IDN host, max-length `tenant_id`, a
differing `protocol_version`, a differing `alg`.
