## 1. What Hugin Agent is

An **outbound-only local bridge daemon** (`hugind`). It installs on a user's
machine behind NAT/firewall, **dials out** over WSS to a cloud orchestrator,
receives commands, runs the local **Claude Code / Codex CLI headlessly**, and
streams normalized results back. No inbound port is ever opened. The cloud relay
is a separate effort; **this repo owns the local side + the shared wire protocol.**

The protocol is a **shared contract** — both `hugind` and the cloud relay import
the same zod SSOT. Changing wire-visible fields requires both sides to agree.

## 4. Locked decisions (do NOT relitigate)

- **EventKind = Option A: core kinds are FROZEN for v1.** Engine-specific events
  use `vendor.<engine>.*` only. A new *core* kind = major bump (v2). This makes
  "unknown core → NACK" (F6) consistent. **Consequence for v1.6:** there is NO
  "deferred final membership" — the core list is final at freeze. You MUST update
  README and PROPOSAL to say so (they currently still say "deferred to adapter
  lock" — that wording is now wrong; see 5e/5f).
- **`agent_id` = per-device**, with a **user 1:N agent** layer:
  `tenant_id > user_id > agent_id (one device/daemon) > engines[]`.
  - "One user, many CLIs" = many machines → many `agent_id` under one `user_id`;
    one machine running claude+codex → one `agent_id`, multiple `capabilities.engines`.
  - `connection_epoch`/lease are scoped **per `agent_id`** → one device reconnecting
    never fences another device of the same user.
  - **Wire does NOT change.** `agent_id` stays per-device (already in `hello`).
    `user_id` is NOT on the wire — it lives in the **pairing record** mapping
    `agent_id → (tenant_id, user_id, key_id)`, is returned in the pairing response,
    and appears only in **audit events**. The signed transcript keeps `tenant_id`
    only (cloud decision); `agent_id` represents the (user, device) pair.

### 5b. EventKind A + F6 rule (`messages.ts` comment + `protocol/README.md`)

`EventKind` already rejects unknown core kinds at parse (closed enum) — that IS F6
enforcement. **Do NOT add a `non-critical` field** (Option A = cores frozen, so the
"ignore non-critical unknown core" branch doesn't exist in v1). Document in
`protocol/README.md` as a **new `## Event kinds` subsection right after `## Versioning`**:
- Unknown **core** kind → receiver NACKs (`invalid_message`); never reinterpret as vendor.
- Unknown **vendor** kind → may be ignored/logged/passed through opaquely.
- **Sender obligation:** a sender emits only core kinds defined at/below the negotiated
  version. (Within frozen-v1 this is automatic — the core list never grows, so it's a
  no-op statement that becomes meaningful only at a future major bump.)
- Note `validateInbound()` can't enforce F6 (parse throws first); F6 lives in the daemon/
  relay's parse-error→nack layer. Say so.

### 5c. Spec edits — auth/pairing (`docs/auth-pairing-spec.md`)

- **F2 server_origin — decide & state the model: REJECT non-canonical input** (don't
  silently normalize). The agent dials a configured URL and signs its canonical origin;
  the verifier reconstructs the same and rejects mismatches. Add normatively:
  production = DNS hostname only (**reject bare IPv4/IPv6**); local-dev exception limited
  to `localhost`/`127.0.0.0/8`/`[::1]` (only there may `ws://` be used); **reject port 0**;
  omit default ports; IPv6 bracket form; reject zone-IDs and percent-encoded authority;
  lowercase even after IDNA punycode.
- **F3 `tenant_id`** — set grammar to `1*128(ALPHA/DIGIT/"-"/"_"/".")` and **update
  §2 table + §11 item 1 from `≤64` to `≤128`** (brief and spec currently disagree). Add
  key-lifetime stability + no silent cross-tenant rebind (re-pairing required).
- **F5 linearizability — make concrete.** The POP MUST linearize: (1) nonce consume
  (atomic check-unspent→mark-spent), (2) `connection_epoch` issuance (strictly monotonic
  **per `agent_id`**), (3) lease ownership (≤1 owner outside rotation overlap, CAS/fencing).
  OK: strong SQL / etcd / Consul / ZK / FoundationDB / Dynamo conditional. NOT OK:
  eventually-consistent cache, per-region Redis, pod-local memory. Note re-pairing (new
  `agent_id`) may reset the epoch namespace. Cross-POP out of MVP scope.
- **Identity model** — add `tenant_id > user_id > agent_id(device) > engines`; resolve the
  open "`agent_id` per-device vs per-host" item (§11 item 5) to **per-device**; describe
  `user_id` flow per §4 (pairing record + pairing response + audit; not on the wire).
- **Remaining ⚙️ open items (§11 items 3/4/6 — rotation grace, pairing-code TTL/device
  limits, audit event schema):** these are operational params, NOT wire/spec freeze
  blockers. Propose strawman values and mark them "post-freeze confirm" in PROPOSAL (5f) so
  they don't block `v1.0.0`.
- **Clarifications:** all base64url unpadded (reject `=`); `server_origin`/`tenant_id` are
  NOT in `hello` (verifier reconstructs from pairing record + trusted connection metadata;
  reject if not reconstructable); transcript `protocol_version` is the exact `hello` value
  (don't verify under a different negotiated version — reject); `challenge_ttl_ms` advisory.

### 5d. NEW — F4 cross-language test vectors

Create **`protocol/v1/gen-vectors.ts`** (a `tsx` generator) that writes
**`protocol/v1/test-vectors.json`**, and a checker invoked from `selftest.ts`.

- Keypair: `crypto.generateKeyPairSync("ed25519")` is random; instead derive from a
  **fixed 32-byte seed** so vectors are reproducible — build the PKCS8 DER from the seed
  (or use `crypto.createPrivateKey` with a fixed seed via the raw Ed25519 import). Record
  `ed25519_seed_hex` + `ed25519_public_hex`.
- Each **positive** vector: `{ challenge_id, nonce_base64url, nonce_raw_hex, agent_id,
  key_id, protocol_version, tenant_id, input_server_origin, canonical_server_origin,
  domain_tag_hex (= LP("hugin-agent/auth/v1")), alg_lp_hex (= LP("ed25519")),
  transcript_hex, ed25519_seed_hex, ed25519_public_hex, expected_signature_base64url }`.
  Transcript per auth-spec §5: `LP(x)=uint32_be(len)||x`, with `nonce_raw` inserted as the
  **raw 32 bytes (NO LP)**. Include `alg_lp_hex` so a Go/Rust impl can verify the `LP("ed25519")`
  bytes + position independently.
- **Positive boundary:** a valid 43-char nonce.
- **Negative vectors (MUST fail, with a `reason` field):** 42- and 44-char nonce; padded
  nonce (`…==`); uppercase host; `:443` present; trailing-dot host; an IDN host;
  max-length (128) `tenant_id` (valid) and 129-char (fail); **non-ASCII `agent_id`** and
  **invalid-charset `agent_id`** (e.g. `agent@host`) — these validate the Codex-B `AuthId`
  fix; **over-long (`>64`) `protocol_version`** — validates Codex-A; a differing
  `protocol_version` (signature mismatch). Drop "differing alg" (alg is `z.literal("ed25519")`,
  schema-rejected — no vector needed).
- `selftest.ts`: load `test-vectors.json`, for positives recompute transcript+verify the
  signature matches, for negatives assert the documented failure (parse reject or verify
  fail). Add these as entries in the existing `checks` array.
